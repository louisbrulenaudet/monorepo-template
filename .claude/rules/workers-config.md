# Wrangler / Config Rules

`wrangler.jsonc` is the **source of truth** for a Worker's bindings, vars, and routes. Two files derive from configuration and must never be hand-edited (they are outputs — see [guardrails.md](guardrails.md)):

- **`worker-configuration.d.ts`** — generated by `wrangler types`. After changing bindings/vars in `wrangler.jsonc`, run **`make types`** to regenerate it. Never edit it by hand.

## Secrets vs vars

- **Secrets** (`SERVICE_*_API_KEY`, `SESSION_SECRET`, `MCP_API_KEY`, `HAIKU_MCP_API_KEY`, `AGENT_API_KEY`) never go in `wrangler.jsonc` `vars`, code, or logs. Local dev: `.dev.vars` (git-ignored; keep `.dev.vars.example` in sync). Prod: `wrangler secret put`.
- **Non-secret config** (`API_ORIGIN`, `*_BASE_URL`, `AI_GATEWAY_ID`, `LOGIN_UI_ORIGIN`, `ENVIRONMENT`) may be plain `vars`. `VITE_API_BASE_URL` for the SPA is build-time and public (not a secret).
- Hand-declared bindings live in `src/types/*.d.ts` (e.g. `oauth-env.d.ts`, `secrets-env.d.ts`); generated ones come from `make types`. When you add a secret, declare it in `secrets-env.d.ts` and add it to `.dev.vars.example` and the setup docs.

## Discipline

- Production route/env changes are checked by the pre-deploy validator (`scripts/validate-production-config.mjs`). Do not weaken it to pass.
- After editing any `wrangler.jsonc`, run `make types` and `make check-types`.
- Workers Cache (`cache.enabled`, per-entrypoint `exports`): see [workers-cache.md](workers-cache.md).
